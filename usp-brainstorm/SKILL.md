# USP Brainstorm — Multi-Model Cross-Validated Pipeline

> **Core Principle**: 行銷話術必須經得起第三方審查。每一句 claim 都要有證據鏈、反證揭露、可量測指標。
> 「吹自己可以，貶低對手不行」是硬規則，沒有例外。

---

## 競品比較硬規則（所有 Phase 都適用）

| # | 規則 | 說明 |
|---|------|------|
| R1 | **只能吹自己** | 可以說「我們達到 X 效能」，不能說「對手的 X 很爛」、「對手不如我們」 |
| R2 | **證據鏈必備** | 每個論述：數據來源 → 測試方法/標準 → 可驗證方式。引用能效數據必須與官方核准標示一致，標明 CNS/ISO 測試標準和條件，不得自行換算 |
| R3 | **反證揭露** | 主動列出「對手在哪些面向可能同等或更強」，不藏弱點 |
| R4 | **可量測** | 優先用數字（省電 X%、噪音 X dB、CSPF X.XX），避免空泛形容詞 |
| R5 | **可驗證** | 消費者或第三方能用什麼方式確認？（能效標示、實測報告、公開規格書） |
| R6 | **第三方可審查** | 論述經得起公平會、消保官、媒體、競品法務檢驗 |
| R7 | **違規即刪** | 任何違反 R1-R12 的話術，直接標記 `VIOLATION` 並從候選清單移除 |
| R8 | **比較基準一致性** | 比較必須是同級產品、使用相同測試標準（CNS/ISO）、相同測試條件。不得以旗艦比入門、以新品比舊款、以不同測試條件比較（公平會比較廣告處理原則第 2/3/4 點） |
| R9 | **禁止選擇性呈現** | 比較表不得只列自家強項而省略競品強項。若列出比較，必須同時呈現競品同等或更強的項目。違反此規則是公平會最常裁罰的類型（LG 洗衣機案 55 萬元、日立冷氣案 60 萬元） |
| R10 | **廣告承諾 = 法律義務** | 話術一旦用於對外廣告（含官網、社群、DM、直播），其內容即構成企業對消費者的法律義務（消保法第 22 條）。「僅供參考」不能免責。消費者可依消保法第 51 條求償最高 5 倍懲罰性賠償 |
| R11 | **能效標示引用規範** | 家電產品（冷氣/洗衣機/冰箱等）的能效數據有法定標示格式。比較表中引用能效數字時不得自行換算、不得脫離 CNS 測試標準（能源管理法第 14 條、能源效率標示管理辦法） |
| R12 | **薦證揭露義務** | 若話術提供給薦證者（網紅/KOL/直播主/經銷商）使用，必須要求揭露品牌合作關係（公平會薦證廣告規範說明）。廣告代理業、媒體業若明知或可得而知廣告不實仍刊登，負連帶責任 |

### 法規參考

- 公平交易法第 21 條（虛偽不實或引人錯誤之表示）：罰鍰 5 萬～2,500 萬元
- 公平交易法第 25 條（欺罔或顯失公平）
- 公平交易委員會對於比較廣告案件之處理原則（8 大禁止行為）
- 消費者保護法第 22 條（廣告承諾義務）、第 51 條（懲罰性賠償最高 5 倍）
- 能源管理法第 14 條 + 能源效率標示管理辦法

## Pipeline 總覽

```
使用者丟概念（產品特性 + 目標客群 + 競品）
    |
Phase 0: 收集輸入 + 任務分類 + 讀 SSOT
    |
Phase 1: Brainstorm（Claude + Codex 雙路互博）
    |
使用者挑選 / 收斂（或 autopilot 自動收斂）
    |
Phase 2: Fact Check（Claude + Codex + Gemini 三路並行）
    |
Phase 3: Adversarial Review（Claude + Codex 雙路並行）
    |                |
    |        P1-DESIGN? ──> 回到 Spec 修改功能需求 ──> 重走 Phase 3
    |
Phase 4: 彙整輸出（依任務類型選擇格式）
    |
（可選）Phase 5: 視覺化（nano-banana 或 code）
```

## Phase 0: 收集輸入

向使用者確認以下資訊（缺什麼問什麼）：

| 項目 | 說明 | 必填 |
|------|------|------|
| **任務類型** | **話術類**（廣告文案、業務話術卡、簡報標語）或 **工具/產品類**（LIFF、app、比較器、UI） | Y |
| **產品** | 型號、品類、關鍵規格 | Y |
| **目標客群** | 誰會買？什麼場景？ | Y |
| **競品** | 要比較哪些品牌/型號？ | 建議填 |
| **SSOT 路徑** | 已存在的產品資料、規格表、審查報告、設計稿的檔案路徑。有 SSOT 就讀 SSOT，不靠 AI 一般知識猜測 | 建議填 |
| **已知賣點** | 使用者已經有想法的角度 | 選填 |
| **禁區** | 不能提的點（例如某個規格不如人） | 選填 |
| **輸出用途** | 簡報 / 話術卡 / 內部提案 / 社群文案 / LIFF UI / 比較器 | Y |
| **執行模式** | **interactive**（每個 Phase 結束等確認）或 **autopilot**（全自動到底） | Y，預設 interactive |

### 任務類型如何影響 Phase 1 題目

| 任務類型 | Phase 1 brainstorm 題目方向 |
|---------|--------------------------|
| **話術類** | 「這個產品有哪些角度可以講？功能面/情感面/對比面/經濟面的話術候選」 |
| **工具/產品類** | 「這個工具還缺什麼功能？消費者/業務用的時候會覺得少什麼？UX 怎麼讓差異化更明顯？」 |

### SSOT 讀取規則

如果使用者提供了 SSOT 路徑：
1. Phase 0 結束前，**必須讀取 SSOT 目錄下所有相關檔案**
2. Phase 1 prompt 中附上 SSOT 摘要作為事實基礎
3. Phase 2 fact check 以 SSOT 為 ground truth，AI 一般知識只做補充
4. 若 SSOT 與 AI 一般知識矛盾，**以 SSOT 為準**

## Phase 1: Brainstorm（雙 AI 互博）

### 為什麼要雙路

單一 AI brainstorm 會有同質化盲區。不同模型的訓練資料和推理風格不同，天然會切出不同角度。

### 執行方式

**同時**發出兩個獨立請求（不讓它們看到彼此的輸出）：

#### 路線 A — Claude Agent

```
Agent(
  subagent_type: "general-purpose",
  prompt: "[依任務類型選擇對應 prompt]

           # 話術類 prompt:
           你是 USP 話術專家。根據以下產品資訊，產出 5-8 個不同角度的話術候選。
           ...（角度：功能面/情感面/對比面/經濟面）

           # 工具/產品類 prompt:
           你是產品設計專家。根據以下工具概念和產品資訊，產出 5-8 個功能差異化建議。
           重點：消費者/業務實際使用時會覺得缺什麼？什麼功能能讓這個工具比競品的工具更有用？
           ...（角度：UX/資訊架構/互動設計/資料呈現/場景覆蓋）

           硬規則：
           - 只能吹自己產品的優勢，不能貶低競品
           - 每個建議必須附上「事實基礎」
           - 標出哪些是硬數據、哪些是主觀感受

           SSOT 摘要：[Phase 0 讀取的 SSOT 內容]
           產品資訊：[Phase 0 收集的資料]"
)
```

#### 路線 B — Codex CLI

```bash
# 語法可能因版本而異，執行前先 codex exec --help 確認
codex exec "[同上 prompt，依任務類型選擇]"
```

### 彙整

收到兩路結果後，Claude 主對話彙整：

```markdown
## Brainstorm 彙整

### 高共識（兩邊都提到的角度）
| # | 角度 | Claude 版 | Codex 版 | 合併建議 |
|---|------|-----------|----------|---------|

### 獨特觀點（只有一邊提到）
| # | 來源 | 角度 | 為什麼值得保留 |
|---|------|------|--------------|

### 矛盾切角（兩邊方向相反）
| # | Claude 說 | Codex 說 | 需要使用者決定 |
|---|-----------|----------|---------------|
```

**Interactive 模式**：等待使用者挑選 / 收斂後，才進入 Phase 2。
**Autopilot 模式**：自動收斂（見下方規則），直接進入 Phase 2。

### Autopilot 收斂規則

| 類別 | 自動處理方式 |
|------|------------|
| 高共識 | 全部保留 |
| 獨特觀點 | 全部保留（獨特才有互博的價值） |
| 矛盾切角 | 取中性版（不暗示貶低競品的那一邊），兩邊都中性則兩個都保留 |

## Phase 2: Fact Check（三路並行交叉查證）

### 為什麼要三路

- **Claude**：邏輯推理強，擅長找論述漏洞
- **Codex**：獨立視角，不同訓練資料
- **Gemini**：有 web search grounding，擅長查即時資料、規格表、公開報告

### 執行方式

對 Phase 1 收斂後的每個候選，**同時**三路查證：

#### 路線 A — Claude Agent

```
Agent(
  subagent_type: "general-purpose",
  prompt: "你是事實查核員。對以下候選逐條驗證：

           對每個 claim：
           1. 事實基礎：這個數字/規格從哪來？來源可信度？
           2. 測試標準：用什麼條件測的？條件合理嗎？
           3. 可驗證性：消費者能自己確認嗎？怎麼確認？
           4. 法規風險：公平交易法、廣告不實、能效標示規範有沒有問題？
           5. 反證：有沒有公開資料能反駁這個 claim？
           6. 比較基準：如果涉及跨品牌比較，是否同級、同標準、同條件？（R8）
           7. 選擇性呈現：是否只列自家強項而省略競品強項？（R9）

           判定等級：
           - VERIFIED：有可靠來源 + 可驗證 + 無明顯反證
           - PARTIAL：部分正確但有條件限制或模糊空間
           - UNVERIFIED：找不到可靠來源，不建議使用
           - VIOLATION：違反競品比較硬規則 R1-R12

           SSOT 參考：[Phase 0 讀取的 SSOT 內容]
           候選清單：[Phase 1 收斂結果]"
)
```

#### 路線 B — Codex CLI

```bash
# 語法可能因版本而異，執行前先 codex exec --help 確認
codex exec "你是獨立事實查核員。對以下候選逐條驗證：
[同上 prompt，含候選清單]"
```

#### 路線 C — Gemini Web Search

```bash
curl -s "https://generativelanguage.googleapis.com/v1beta/models/gemini-2.0-flash:generateContent?key=$GEMINI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "contents": [{"parts": [{"text": "你是事實查核員。請用 web search 驗證以下產品候選的事實基礎。\n\n對每個 claim：\n1. 搜尋相關規格表、官方資料、第三方測試報告\n2. 確認數字是否正確\n3. 找出可能的反證\n4. 標記來源 URL\n\n候選清單：[Phase 1 收斂結果]"}]}],
    "tools": [{"google_search": {}}]
  }'
```

**Gemini 錯誤處理**：如果 API 回傳空值、錯誤、或 timeout，標記為「Gemini: FAILED」並觸發降級規則。

### 彙整

```markdown
## Fact Check 彙整

| # | 候選 | Claude | Codex | Gemini | 共識判定 | 證據鏈 | 反證 |
|---|------|--------|-------|--------|---------|--------|------|

### 有效路數：[3/3] / [2/3] / [1/3]
```

### 判定規則

| 有效路數 | 處理 |
|---------|------|
| **3 路有效** | 正常判定 |
| **2 路有效** | 標記 `REDUCED COVERAGE`，判定標準不變 |
| **1 路有效** | 標記 `SINGLE SOURCE`，所有判定自動降一級（VERIFIED → PARTIAL, PARTIAL → UNVERIFIED） |
| **0 路有效** | `FACT CHECK FAILED`，不進入 Phase 3，要求使用者手動補充事實基礎 |

### 判定組合

- 多數 VERIFIED → **PASS**，可進入 Phase 3
- 多數 PARTIAL → **CONDITIONAL PASS**，需附註條件
- 任一路 UNVERIFIED → **NEEDS REVIEW**，由使用者決定（autopilot: 自動移除）
- 任一路 VIOLATION → **自動移除**，不進入 Phase 3

## Phase 3: Adversarial Review（雙路壓力測試）

### 審查維度

| 維度 | 問什麼 |
|------|--------|
| **事實可攻擊性** | 對手能拿公開數據打臉這句話嗎？ |
| **邏輯漏洞** | 因果關係成立嗎？是不是偷換概念？ |
| **法規風險** | 公平交易法 21/25 條、比較廣告處理原則、消保法 22/51 條、能效標示管理辦法 |
| **競品反擊** | 對手的同級產品能講一樣的話嗎？如果能，這就不是 USP |
| **消費者感知** | 聽起來厲害但消費者其實無感？ |
| **數字陷阱** | 測試條件是否刻意選擇有利場景？ |
| **時效性** | 這個優勢能維持多久？對手下一代會追上嗎？ |
| **選擇性呈現** | 比較表是否只列自家強項？（R9，公平會最常罰） |

### 執行方式

**同時**發出兩路：

#### 路線 A — Claude Agent

```
Agent(
  subagent_type: "general-purpose",
  prompt: "你是競品的行銷主管 + 消保官 + 公平會調查員。找出以下候選的每一個可攻擊點。

           對每個候選，依序檢查 8 個維度：
           1. 事實可攻擊性
           2. 邏輯漏洞
           3. 法規風險（特別注意公平會比較廣告處理原則第 6 點：禁止以局部優勢暗示全面優勢）
           4. 競品反擊
           5. 消費者感知
           6. 數字陷阱
           7. 時效性
           8. 選擇性呈現（R9）

           嚴重度：
           - P0 CRITICAL：直接違法或事實錯誤，必須刪除
           - P1 MAJOR：有明顯攻擊面，需要修改措辭或補充條件
           - P1-DESIGN：攻擊面涉及功能設計層面（不是措辭問題），必須回到 spec 修改功能需求
           - P2 MINOR：小風險，可接受但建議改善

           候選清單 + 證據鏈：[Phase 2 PASS 的結果]"
)
```

#### 路線 B — Codex CLI

```bash
# 語法可能因版本而異，執行前先 codex exec --help 確認
codex exec "你是競品的行銷主管 + 消保官。找出以下候選的每一個可攻擊點。
[同上 prompt，含候選清單 + 證據鏈]"
```

### 彙整與判定

```markdown
## Adversarial Review 彙整

### 共識問題（Claude + Codex 都找到）
| # | 候選 | 嚴重度 | 攻擊面 | 建議修改 |
|---|------|--------|--------|---------|

### Claude 獨有 Findings
| # | 候選 | 嚴重度 | 攻擊面 | 交叉驗證 |
|---|------|--------|--------|---------|

### Codex 獨有 Findings
| # | 候選 | 嚴重度 | 攻擊面 | 交叉驗證 |
|---|------|--------|--------|---------|
```

### 判定規則

| 嚴重度 | 處理 |
|--------|------|
| **P0** | 直接移除或徹底改寫 |
| **P1** | 修改措辭、補充限制條件、加註但書 |
| **P1-DESIGN** | **回到 spec 修改功能需求**，然後重走 Phase 3 驗證修改後的設計。不能只用 disclaimer 打發 |
| **P2** | 記入殘餘風險，可保留 |

### P1-DESIGN 閉環規則

如果 adversarial review 發現的問題本質上是**功能設計**的問題（例如「評分系統不透明」「缺少某個維度」「資訊架構讓使用者困惑」），不能只改措辭或加 disclaimer：

1. 標記為 `P1-DESIGN`
2. 回到 spec / Phase 0，新增或修改功能需求
3. 重新進入 Phase 3 驗證修改後的設計
4. 直到沒有新的 P1-DESIGN 為止

**判斷標準**：如果修復方式是「加一行 disclaimer」或「換個用詞」，那就是 P1（措辭問題）。如果修復方式是「需要加一個新頁面」「需要改資料結構」「需要新增互動功能」，那就是 P1-DESIGN。

## Phase 4: 最終輸出

通過三層審查的候選，依 **Phase 0 的任務類型和輸出用途**整理成最終交付物。

### 輸出格式依用途選擇

| 輸出用途 | 格式 |
|---------|------|
| 廣告文案 / 簡報標語 | 話術卡（一句話版 + 完整版 + 證據鏈） |
| 業務話術卡 | 話術卡 + 反證揭露 + 消費者可能的反問 Q&A |
| LIFF / web / 比較器 | UI 功能建議清單（功能描述 + 實作建議 + 護欄） |
| 內部提案 | 話術卡 + 法規風險摘要 + 殘餘風險 |
| 社群文案 | 話術卡（短版）+ R12 薦證揭露提醒 |

### 話術卡格式（話術類用）

```markdown
## USP 話術卡：[產品型號]

### 話術 #N
- **一句話版**：[30 字以內，可直接用於廣告/簡報]
- **完整版**：[100 字以內，含數字和條件]
- **證據鏈**：
  - 數據來源：[官方規格 / 第三方測試 / 能效標示]
  - 測試標準：[CNS / ISO / 自有測試，測試條件]
  - 驗證方式：[消費者可以怎麼確認]
- **已知限制**：[在什麼條件下這句話不成立]
- **反證揭露**：[對手在哪些面向可能同等或更強]
- **法律義務提醒**：此話術若用於對外廣告，即構成企業對消費者之義務（消保法第 22 條）
- **審查結果**：Fact Check [PASS/CONDITIONAL] | Adversarial [P2 殘餘風險/CLEAN]
```

## Phase 5: 視覺化（Decision Gate）

### 觸發判斷（不是「可選」，是明確的 decision gate）

| 條件 | 用什麼 | 理由 |
|------|--------|------|
| 輸出用途是 **LIFF / web** + 需要圖表 | **Code**（SVG / Canvas） | 互動圖表需要程式繪製，nano-banana 畫不出互動 |
| 輸出用途是 **簡報 / 文件** + 需要圖表 | **nano-banana** | 靜態圖嵌入 PPTX/DOCX |
| 需要**概念示意圖**（「AI 氣流怎麼運作」） | **nano-banana** | 這不是數據圖表，是概念視覺化 |
| 需要**場景情境圖**（「回家的舒適感」） | **nano-banana** | 生成式 AI 圖片 |
| 不需要視覺化 | **跳過** | — |

## 規則總結

1. **競品硬規則 R1-R12 在所有 Phase 都適用**，任何產出違反即標記 VIOLATION
2. **Phase 0 必須先分類任務類型**（話術 vs 工具），不同類型用不同 brainstorm 題目
3. **Phase 0 如果有 SSOT，必須先讀**，不靠 AI 一般知識猜測
4. Phase 1 的 Claude 和 Codex **必須平行執行、互不可見**
5. Phase 2 有效路數不足時，**必須標記降級**，不能冒充三路結論
6. Phase 3 的 P1-DESIGN **必須回到 spec 修改功能需求**，不能只改文案
7. **Interactive 模式**：每個 Phase 結束等使用者確認。**Autopilot 模式**：依收斂規則自動進入下一個 Phase
8. Phase 4 輸出格式**依任務類型和輸出用途選擇**，不是只有話術卡一種
9. Phase 5 視覺化有明確的 decision gate，不是「可選」
